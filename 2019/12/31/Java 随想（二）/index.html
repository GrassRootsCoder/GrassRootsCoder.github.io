白天为生存，晚上谋发展

一天由二十四个小时组成，时间是有限的，我们白天会花费大量时间用于工作换取金钱为了生存，夜晚则会被睡眠占用大量时间，一天当中给留我们自己支配的时间已所剩无几，工作日是这样度过。而周末会有加班、应酬等情况，这也占用周末大量时间，还剩下一些时间也大概率是用来休息，缓解工作日带来的疲劳。感受一下，一周大部分时间都用在为了生存上，这样的一周为自己又干了什么呢。

当工作或者生存占用到个人大量时间后，很少有时间和精力来思考自己的问题（没错，这就是社畜），没有办法发展自己的兴趣、爱好和享受生活。想要跳出这个圈就必须得抽出属于自己的时间，工作日的晚上下班后到睡觉前这段时间，周末也能抽出一些。

为什么晚上谋发展对于个人来说那么重要？我自己感觉现在的公司无法给员工充分的依赖（不像以前在一家公司可以待很久），所以员工必须对自己的人生负责，要时刻关注自己。（否则，就像我一样随着工作年限增加，感受到时间流逝越来越快，除了工作上有所获得，自己本身好像没有得到些什么，也可谓“碌碌无为又一年”，逃。

2.事务以及隔离级别

数据库的日常使用离不开并发操作，在并发状态下数据库使用数据库事务来确保数据的正确，数据库事务主要包含四个特性，简称为：ACID特性，即：

  1. 原子性（Atomicity）：事务作为一个整体被执行，所包含的所有操作要么都执行，要么都不执行。
  2. 一致性（Consistency）：事务要确保数据库的状态从一个一致状态转变为另一个一致状态。
  3. 隔离性（Isolation）：多个事务并发执行时，一个事务的执行不影响其他事务的执行。
  4. 持久性（Durability）：已提交的事务对数据库的修改应永远保存在数据库中。

从上述描述中可以知道事务是如何保证数据正确的，接下来用 SQL 并发方式展示事务的几种隔离等级，隔离等级越高就表示出现异常的频率就越低，同时效率就越慢。

事务隔离级别一共只有四种级别，即：

  1. 可串行化（Serializable）：最高隔离级别，不会发生任何读取错误现象，但是会增大死锁概率。
  2. 可重复读（Repeatable Read）：会添加读锁（Read Locks）和写锁（Write Locks）一直到事务结束，但没有范围锁，有可能会发生“幻读”现象。（幻读：在一次事务中同样的查询语句能够读取到不同的结果集）
  3. 提交读（Read Committed）：该级别添加的读锁（Read Locks）在 SELECT 语句执行完以后立即释放，有可能会发生“不可重复读”现象。（在一次事务中读取同一条数据两次，会出现不同的数值）
  4. 未提交读（Read Uncommitted）：最低隔离级别，可能会发生“脏读”现象。（读取到没有提交的数据）

不同的数据库默认隔离级别不同，MySQL 默认隔离级别是可重复读。

3.有关MySQL

MySQL 包含的知识和技术太多了，仅仅介绍几个点是不行的，甚至是看几本书也无法掌握 MySQL。不同的技术人员对于数据库的掌握程度不同，DBA 需要多读很多书籍，但 Java 开发不用看那么多书。

这里推荐两本书籍：《MySQL 必知必会》算是 SQL 入门级别的书，只是一个小册子，平常可以没事翻翻看，熟悉 SQL 的基本使用。另外一个是极客时间专栏《MySQL 实战45讲》，里面包含了很多常使用到的知识以及面试题，对于开发人员算是有难度的，可以慢慢读完。

最后还有一本《高性能 MySQL》，业界给这本书的评价非常高，之前看同事买了一本，我偶尔拿来翻翻，感觉内容不太适合开发去看，同时书籍很厚，估计会让大部分人打退堂鼓，所以这里就不推荐了，如果有兴趣转型 DBA 的话可以去看看。

2.如何实现分布式锁

Redis 实现分布式锁的原理是：

SETNX key value

SETNX 这个命令全称是：SET if Not eXists，如果不存在则 set

成功返回1，失败返回0

设计锁的思路：必须有获取锁和释放锁的动作，否则会出现死锁或者并发操作现象。

  1. 通过 SETNX 命令获取锁，如果返回1则说明获取锁成功，返回0则说明失败。
  2. 获取锁的线程在处理完业务之后要释放锁（DEL）。
  3. 需要设置锁过期时间（EXPIRE），在处理超时或者发生错误时能够不出现死锁现象。
  4. 锁过期时间不能太短，这样有可能业务还没有处理完成；不能太长，会导致资源都处于等待获取锁状态。

RedisDistributedLock 注解，用于给指定方法添加分布式锁。

DistributedLockAOP 使用 @Around 注解，在方法执行前获取锁，如果获取失败则不执行方法（具体逻辑可以根据业务来修改），最后释放锁。

最后在想要添加分布式锁的方法上加上注解即可。可以使用多线程或者集群方式来验证分布式锁是否成功。
