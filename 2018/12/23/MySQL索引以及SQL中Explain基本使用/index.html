MySQL 索引以及 SQL 中 Explain 基本使用

Posted on 2018-12-23 | Post modified: 2018-12-23 | In Database

SQL 语句中对表的操作分为四类：Create，Read，Update，Delete（中文：增删改查；英文：CRUD）。其中，查询语句是占实际使用比率最多的类型，既然这么重要就有必要来学习查询语句相关使用和优化，这个点在面试中也经常出现。

针对查询语句主要有两个点：索引和 EXPLAIN。索引是优化查询效率，将全表扫描变为扫描索引，通过索引提升执行效率；EXPLAIN 是分析查询语句是否使用到索引，程序员修改 SQL 语句的依据。

接下来就介绍索引是什么，以及如何使用 EXPLAIN。

MySQL 索引相关内容

相信大家对于索引（index）早已不陌生，这里也不过多解释索引的定义，就使用一个简单的比喻来理解索引吧：“在摆放几本书的书桌上找到一本书非常容易，一眼望过便能找到；在摆放几排书的书架上找到一本书有些困难，需要找到每排书的分类，然后从某一排书中找到；在藏书如海的图书馆中找到一本书由如海底捞针，需要找到书籍的多层分类，然后对应书架、对应排号和对应位置。索引就是用于快速查找书籍的分类目录，可以将书籍所在位置快速确定，不用一点一点去寻找。”如果书籍（数据）非常少，一眼望过便能找到，则不需要书籍目录，反而会浪费查找时间（MySQL 中有时用不上索引就是如此，扫描全表效率比索引快）。如果书籍非常多，比图书馆还多数倍以上，索引还会管用么（这个问题就需要大数据去解决了，这里不涉及）？

索引实现算法

MySQL 常见索引实现算法有两种：BTree 和 Hash，默认表引擎是 InnoDB，使用 BTree 结构。

BTree：可以用于范围（不限于 =, >, <, >=, <= 和 between）和 like 操作符查询，只要查询条件不是以 % 开头的常量（比如 WHERE username LIKE ‘%Jack’）。

Hash：仅用于等于（=）查询，不支持排序和最左匹配规则。是一次性定位查询，不需要像 BTree 那样遍历树节点，查询效率高（仅限于没有重复键值的情况下，否则会因为哈希碰撞问题而降低效率）。

PS：注意 MySQL 表引擎，不同引擎使用的索引算法不同。对于 BTree 和 Hash 算法不了解的，可以先去看看相关数据结构资料。

索引类型

MySQL 常见索引类型有三种：主键索引、唯一索引和普通索引。（组合索引：多列组成的索引）

主键索引：数据列值不能重复，不能为null。一张表只能有一个主键索引。

唯一索引：数据列值不能重复，可以为null。一张表可以有多个唯一索引，也可以由多列组成一个唯一索引。

普通索引：对于数据列值没有限制。一张表可以有多个普通索引，也可以由多列组成一个普通索引。

当我们使用组合索引时，要遵循“最左前缀”原则，具体含义是：WHERE 条件子句中最频繁使用的一列放在条件最左边。比如：

聚簇索引 和 非聚簇索引：区别在于叶子节点是否存放一整行记录。（不是索引类型，是索引数据存放）

InnoDB 引擎主键索引是使用聚簇索引，MyISAM 引擎不管是主键索引还是其他索引都是用非聚簇索引。（不涉及这两个索引优缺点，感兴趣的可以自行搜索了解）

索引相关面试内容

1.如何知道哪些 SQL 需要优化，怎么优化？

可以开启 MySQL 慢查询功能，设置 SQL 执行时间和是否使用索引等参数，找到需要优化的 SQL。也可以根据线上使用人员或者测试人员反馈结果，找到导致接口响应时间长的 SQL。

可以使用 EXPLAIN 关键字分析 SQL，合理使用和创建索引，优化 SQL 中 WHERE 条件子句内容。

2.如何设计索引才能保证其使用效率？

a.给 WHERE 条件中使用最多到列创建索引。

b.使用短索引（给字符串索引设置前缀长度），不要过度创建索引。

c.表数据较少的情况下，可以不创建索引，直接全表扫描。

3.为什么创建了索引，SQL 也写对了 WHERE 条件，却还没有使用上索引呢？

一条 SQL 语句执行方案有多种，具体执行哪一种方案，由优化器来进行选择，方案过程如下：

a.根据 WHERE 条件找出所有可能使用到的索引 。

b.计算全表扫描的代价 。

c.计算使用不同索引执行查询的代价 。

d.对比各种执行方案的代价，找出成本最低的那一个。

因此，没有使用索引并不是 SQL 写错，而是全表扫描效率最高。

EXPLAIN 是如何使用的

前文提到过可以使用 EXPLAIN 关键字对 SQL 进行分析，看有没有使用到索引，一般来说工作中就是用这样的方式来修改 WHERE 条件的，接下来介绍使用方法。

首先创建两张表以及插入数据，用于解释 EXPLAIN 使用方法。

EXPLAIN 字段解析

  1. id：SELECT 查询语句序列号。
  2. select_type：查询类型。
  3. table：查询表名。
  4. partitions：匹配分区。
  5. type：查询连接类型。
  6. possible_keys：查询可能使用到的索引。
  7. key：查询实际使用到的索引。
  8. key_len：索引使用到的字节数。
  9. ref：索引使用到的某一列或者常数。
  10. rows：估计查询共扫描了多少行。
  11. filtered：查询条件所过滤掉的数据的百分比。
  12. Extra：额外信息。

EXPLAIN 字段含义如上，接下来对每个字段重点解析一下。

id

该字段表示查询语句的序列号，一般在子查询和连接查询情况下会有多个id，也同时表示 SELECT 语句执行顺序。如果 id 相同，则由上往下顺序执行；如果 id 不相同，则由大到小顺序执行。

参考下面的例子，先执行 3.row 语句，也就是子查询，接着执行 1.row，这个是左连接查询，最后执行 2.row，查询出我们这次想要的结果。

select_type

该字段表示查询类型，主要有以下几种类型：

  • SIMPLE：查询语句不包含子查询（SUBQUERY）或者联合查询（UNION），最简单查询类型。
  • PRIMARY：查询语句包含子查询或者联合查询，是最外层查询语句。
  • SUBQUERY：查询语句是子查询。
  • UNION：查询语句是联合查询。
  • UNION RESULT：联合查询结果集。

第一句 SQL 是单表查询，没有包含子查询或者联合查询。

第二句 SQL 是子查询，可以看到 SUBQUERY 类型，因为包含子查询，所以出现 PRIMARY 类型。

第三句 SQL 是联合查询，可以看到 UNION 类型，同时联合查询是将多个 SQL 结果拼接在一起展示，所以多出一个 UNION RESULT 类型。

table

该字段就很好理解，如果没有联合查询或者设置别名，就展示原有表名，否则展示联合查询结果集表名或者别名。

partitions

如果表存在分区，这里则展示分区信息。（可以自行 Google 了解一下，涉及 MySQL 数据分区）

type

该字段表示查询连接类型，可以根据结果判断查询效率，主要包含以下几种类型（执行效率从上往下依次递减）：

  1. system：表中只包含一条数据。这个是特殊 const 类型，作为了解即可。
  2. const：查询只返回一行结果集，用于主键索引或者唯一索引。因为某一行记录在表中是唯一的，因此查询效率很高。
  3. eq_ref：用于多表联查，某一张表的查询结果都只能匹配到另外一张表的一行结果，常用于主键索引或者唯一索引。
  4. ref：查询结果或者 WHERE 条件可以匹配到某一张表的多行结果，或者使用到最左前缀索引规则，常用于非主键索引或者唯一索引。
  5. range：查询条件是范围查询，比如：IN(), <=, >=, between 等操作符。
  6. index：全扫描索引，没有减少扫描范围。
  7. ALL：全扫描表，没有使用到索引。

第一句 SQL 是直接根据 id 进行查询（主键查询），由于主键索引具有唯一性质，因此是 const 类型。

第二句 SQL 是左连接查询，game_info 表中 machine_info_id 对应 game_machine_info 表中 id 字段，每个 machine_info_id 字段内容都只能对应 game_machine_info 表中一行记录。

第三句 SQL 使用到最左前缀索引规则。

第四句 SQL 根据 id 进行范围查询。

第五句 SQL 直接扫描所有索引，没有对索引的扫描范围进行优化。

第六句 SQL 是使用到没有索引的字段进行条件过滤，因此全表扫描。

possible_keys

该字段只是展示 SQL 执行时可能使用到的索引，会是 NULL，一个或者多个，仅限于可能使用到，实际有没有使用索引需要看 key 字段。

key

该字段表示 SQL 执行时实际使用到的索引。

ken_len

该字段表示具体使用到索引的字节数，可以用来判断具体哪些字段被索引使用了。（需要知道列的数据类型占多少字节，计算出最后结果，感兴趣的同学可以自行 Google）

ref

该字段表示索引的哪一列或者常量被用于索引。

下面这个 SQL 表示 demo.g.machine_info_id 字段被用于查询 game_machine_info 表 id 字段。

rows

该字段表示 SQL 语句查询到结果需要扫描的行数范围，原则上来说，这个数值越少越好。

filtered

该字段表示存储引擎返回的数据在服务层过滤后，能够满足查询条件的数据。

Extra

该字段包含 SQL 语句执行额外信息，一般有以下几种类型：

  • Using index：使用到索引。（仅仅只是使用到索引，至于索引范围需要看其他字段）
  • Using where：使用到条件。
  • Using temporary：使用到临时表。（一般在多表查询的时候会出现，如果看到这个信息可以对 SQL 进行优化，尽量不要使用到临时表）
  • Using filesort：使用到外部排序。（一般来说是 ORDER BY 排序字段没有按照索引顺序进行排序，这里会非常消耗 CPU 性能，需要立即进行优化）

附加信息补充 SQL 语句可以优化的点，比如：temporary 和 filesort，一个是使用了临时表，一个是排序没有按照索引顺序来排序。
